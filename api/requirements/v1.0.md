## 1. Project Overview

DevFolio CMS is a multi-user backend platform where software developers register an account, manage their professional portfolio data (projects, skills, work experience), and expose that data through a public, unauthenticated API. The public API is consumed by a developer's own external portfolio website or any other client they choose to point at it.

The backend is responsible for:

- Developer account management and authentication
- Full CRUD operations for all portfolio entities
- Serving a clean, public, read-only API scoped by username
- Handling image uploads to Cloudflare R2 via presigned URLs
- Sending transactional emails via Nodemailer + SMTP
- Managing a shared, admin-governed technology lookup list

There is no API key system. The public API requires no authentication. Draft and archived content is never reachable from outside the authenticated dashboard.

## 2. Tech Stack & Architecture

### 2.1 Core

| Concern          | Choice                              |
| ---------------- | ----------------------------------- |
| Runtime          | Node.js                             |
| Framework        | NestJS (TypeScript)                 |
| ORM              | Prisma                              |
| Database         | PostgreSQL via Neon (serverless)    |
| Object Storage   | Cloudflare R2                       |
| Email            | Nodemailer + SMTP                   |
| Authentication   | JWT (access token) + refresh token  |
| Password Hashing | bcrypt                              |
| Validation       | class-validator + class-transformer |
| Rate Limiting    | @nestjs/throttler                   |

### 2.2 Repository Structure

The backend lives in an `api/` directory inside a Bun-based monorepo managed by Turborepo:

```
root/
├── api/          ← NestJS backend (this document)
├── client/       ← Next.js frontend (separate requirements)
├── packages/     ← Shared TypeScript types/configs
├── package.json  ← Workspace root
└── turbo.json
```

### 2.3 Architecture Principles

- **Modular structure:** each entity (users, projects, skills, etc.) lives in its own NestJS module with its own controller, service, and DTOs.
- **No business logic in controllers:** controllers handle HTTP concerns only; all logic lives in services.
- **No raw SQL:** all database access goes through Prisma.
- **Soft deletes for users and projects:** records are never hard-deleted; a `deleted_at` timestamp is set instead, and all queries filter on `deleted_at IS NULL`.
- **Presigned uploads:** the backend never proxies image bytes. It issues presigned R2 PUT URLs; the client uploads directly to R2.
- **Global exception filter:** all unhandled errors return a consistent JSON error shape.
- **Global validation pipe:** all incoming request bodies are validated via class-validator DTOs before reaching the controller.

## 3. Database Schema

All tables use UUID primary keys generated by the application layer (not the database). All timestamps are stored in UTC.

### 3.1 users

```prisma
model User {
  id           String    @id @default(uuid())
  email        String    @unique
  passwordHash String    @map("password_hash")
  username     String    @unique
  fullName     String?   @map("full_name")
  avatarUrl    String?   @map("avatar_url")
  bio          String?

  createdAt    DateTime  @default(now()) @map("created_at")
  updatedAt    DateTime  @updatedAt @map("updated_at")
  deletedAt    DateTime? @map("deleted_at")

  projects        Project[]
  skills          UserSkill[]
  experience      Experience[]
  refreshTokens   RefreshToken[]
  passwordResets  PasswordReset[]

  @@map("users")
}
```

**Constraints:**

- `email` — unique, required, must be a valid email format
- `username` — unique, required, 3–30 characters, lowercase alphanumeric and hyphens only, cannot start or end with a hyphen, cannot be a reserved word (see Section 12.3)
- `passwordHash` — never exposed via any API response
- `deletedAt` — all queries must filter `deletedAt IS NULL` unless explicitly performing a recovery operation

### 3.2 refresh_tokens

```prisma
model RefreshToken {
  id        String    @id @default(uuid())
  userId    String    @map("user_id")
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  tokenHash String    @unique @map("token_hash")
  expiresAt DateTime  @map("expires_at")
  revokedAt DateTime? @map("revoked_at")
  createdAt DateTime  @default(now()) @map("created_at")

  @@map("refresh_tokens")
}
```

**Notes:**

- Refresh tokens are stored as bcrypt hashes, never plaintext
- Revoked tokens are soft-revoked (`revokedAt` set), not deleted, for audit purposes
- Expired and revoked tokens should be purged by a scheduled cleanup job (v1: manual or cron, not a hard requirement)

### 3.3 password_resets

```prisma
model PasswordReset {
  id        String    @id @default(uuid())
  userId    String    @map("user_id")
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  tokenHash String    @unique @map("token_hash")
  expiresAt DateTime  @map("expires_at")
  usedAt    DateTime? @map("used_at")
  createdAt DateTime  @default(now()) @map("created_at")

  @@map("password_resets")
}
```

**Notes:**

- Reset tokens expire after 1 hour
- Tokens are single-use: `usedAt` is set on redemption and the token is rejected on any subsequent use
- Token is stored as a SHA-256 hash; the plaintext token is sent to the user's email only and never persisted

### 3.4 projects

```prisma
model Project {
  id               String        @id @default(uuid())
  userId           String        @map("user_id")
  user             User          @relation(fields: [userId], references: [id], onDelete: Cascade)

  title            String
  slug             String
  shortDescription String?       @map("short_description")
  longDescription  String?       @map("long_description")
  status           ProjectStatus @default(draft)
  isFeatured       Boolean       @default(false) @map("is_featured")
  displayOrder     Int           @default(0) @map("display_order")
  projectType      ProjectType?  @map("project_type")
  liveUrl          String?       @map("live_url")
  repoUrl          String?       @map("repo_url")
  startedAt        DateTime?     @map("started_at")
  completedAt      DateTime?     @map("completed_at")

  createdAt        DateTime      @default(now()) @map("created_at")
  updatedAt        DateTime      @updatedAt @map("updated_at")
  deletedAt        DateTime?     @map("deleted_at")

  techStack        ProjectTechStack[]
  images           ProjectImage[]
  links            ProjectLink[]

  @@unique([userId, slug])
  @@index([userId, status, deletedAt])
  @@map("projects")
}

enum ProjectStatus {
  draft
  published
  archived
}

enum ProjectType {
  web
  mobile
  api
  library
  other
}
```

**Constraints:**

- `slug` — unique per `userId`, 3–60 characters, lowercase alphanumeric and hyphens only, auto-generated from `title` on creation but independently editable thereafter
- `shortDescription` — max 300 characters
- `longDescription` — markdown string, no enforced max length in the database but the API should enforce a reasonable limit (see Section 12)
- `liveUrl` and `repoUrl` — must be valid URLs if provided
- `startedAt` — if both `startedAt` and `completedAt` are provided, `startedAt` must be before `completedAt`

## 5. Authentication & Security

### 5.1 Registration Flow

1. Validate the request body via `RegisterDto`
2. Check `email` is not already registered (case-insensitive) — return `409 Conflict` if taken
3. Check `username` is not already taken (case-insensitive) — return `409 Conflict` if taken
4. Validate `username` against allowed character rules and reserved word list (Section 12.3)
5. Hash the password using `bcrypt` with a cost factor of **12**
6. Create the user record
7. Return the user's public profile (no tokens — require explicit login after registration)

### 5.2 Login Flow

1. Find user by email (case-insensitive) — return `401 Unauthorized` with a generic message if not found (do not reveal whether the email exists)
2. Check `deletedAt IS NULL` — return `401` if account is soft-deleted
3. Compare submitted password against stored hash using `bcrypt.compare`
4. On success, generate:
   - **Access token:** JWT signed with `JWT_SECRET`, payload `{ sub: userId, username }`, expires in **15 minutes**
   - **Refresh token:** cryptographically random string (32 bytes, hex-encoded), hashed with bcrypt (cost 10) and stored in `refresh_tokens`, expires in **14 days**
5. Return both tokens in the response body

### 5.3 Token Refresh Flow

1. Accept the refresh token from the request body
2. Hash the submitted token and look up a matching, non-revoked, non-expired `RefreshToken` record
3. Verify the associated user exists and is not soft-deleted
4. **Rotate the token:** revoke the used refresh token and issue a new one
5. Issue a new access token
6. Return both new tokens

### 5.4 Logout Flow

1. Require a valid access token
2. Accept the current refresh token from the request body
3. Set `revokedAt` on the matching `RefreshToken` record
4. Return `200 OK`

### 5.5 Forgot Password Flow

1. Accept an email address
2. Look up the user by email — **always return the same generic success response regardless of whether the email is registered** (prevents email enumeration)
3. If the user exists and is not soft-deleted:
   a. Invalidate any existing unused password reset tokens for this user by setting `usedAt = now()`
   b. Generate a cryptographically random reset token (32 bytes, hex-encoded)
   c. Store a SHA-256 hash of the token in `password_resets` with `expiresAt = now() + 1 hour`
   d. Send the reset email via Nodemailer (Section 8)
4. Return `200 OK` with a generic message: `"If that email is registered, you will receive a reset link shortly"`

### 5.6 Reset Password Flow

1. Accept `token` and `newPassword` from the request body
2. Hash the submitted token with SHA-256 and look up a matching `PasswordReset` record
3. Reject if: token not found, `usedAt` is already set, or `expiresAt` is in the past — return `400 Bad Request` with a message indicating the link is invalid or expired
4. Validate `newPassword` against password rules (Section 12.1)
5. Hash the new password with bcrypt (cost 12)
6. Update `user.passwordHash`
7. Mark the reset token as used: `usedAt = now()`
8. Revoke all existing refresh tokens for this user (force re-login everywhere)
9. Return `200 OK`

### 5.7 Password Rules

- Minimum 8 characters
- At least one uppercase letter
- At least one lowercase letter
- At least one number
- Maximum 72 characters (bcrypt limit)

### 5.8 Authorization

- All authenticated endpoints use a `JwtAuthGuard` that validates the access token from the `Authorization: Bearer <token>` header
- Controllers use a `@CurrentUser()` decorator to extract the authenticated user's `userId` from the JWT payload
- All service methods that operate on owned resources (projects, skills, experience) must verify the `userId` from the token matches the resource's `userId` before proceeding — return `403 Forbidden` if not
- There is no role-based access control beyond the implicit admin guard described in Section 6.9

## 6. API Endpoints

### Response envelope

All responses follow this shape:

**Success:**

```json
{
  "success": true,
  "data": {}
}
```

**Error:**

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Human-readable message",
    "details": []
  }
}
```

**POST /tech-stack/suggest:** creates a new entry with `status = pending`. If a tech with the same name already exists (case-insensitive, any status), return `409 Conflict` with a message like "This technology already exists or has been suggested."

### 6.10 Upload Endpoints

| Method | Path               | Auth | Description                    |
| ------ | ------------------ | ---- | ------------------------------ |
| POST   | `/uploads/presign` | JWT  | Request a presigned R2 PUT URL |

**POST /uploads/presign request body:**

```json
{
  "filename": "project-screenshot.png",
  "contentType": "image/png",
  "uploadType": "project_image" | "avatar"
}
```

**POST /uploads/presign response:**

```json
{
  "success": true,
  "data": {
    "uploadUrl": "https://...",
    "publicUrl": "https://cdn.yourdomain.com/...",
    "key": "images/userId/uuid.png",
    "expiresIn": 300
  }
}
```

**Validation before issuing URL:**

- `contentType` must be one of: `image/jpeg`, `image/png`, `image/webp`
- File type is inferred from `contentType` only — do not trust the filename extension alone
- Presigned URL expires in **5 minutes**
- After upload, the client calls the relevant endpoint (e.g. `POST /me/projects/:id/images`) with the returned `publicUrl` to persist the record

## 7. File Upload (Cloudflare R2)

### 7.1 Upload Flow

```
Client                    NestJS API                    Cloudflare R2
  |                           |                               |
  |-- POST /uploads/presign ->|                               |
  |   { filename, contentType,|                               |
  |     uploadType }          |                               |
  |                           |-- validate contentType        |
  |                           |-- generate object key        |
  |                           |-- generate presigned PUT URL->|
  |                           |<-- presigned URL ------------|
  |<-- { uploadUrl, publicUrl,|                               |
  |      key, expiresIn } ----|                               |
  |                           |                               |
  |-- PUT image bytes --------|---------------->  R2 bucket  |
  |<-- 200 OK ----------------|-----------------|            |
  |                           |                               |
  |-- POST /me/projects/:id/images                           |
  |   { url: publicUrl, ... } |                               |
  |<-- 201 Created ------------|                               |
```

### 7.2 Object Key Naming

```
images/{userId}/{uploadType}/{uuid}.{ext}
```

Examples:

```
images/abc123/project_image/f47ac10b.png
images/abc123/avatar/9b2d4e1a.webp
```

### 7.3 File Constraints

| Property                   | Constraint                              |
| -------------------------- | --------------------------------------- |
| Allowed content types      | `image/jpeg`, `image/png`, `image/webp` |
| Maximum file size          | **5 MB**                                |
| Maximum images per project | **10**                                  |
| Presigned URL expiry       | **5 minutes**                           |

The file size limit of 5 MB is enforced via the R2 presigned URL's `Content-Length` condition — include this condition when generating the presigned URL so R2 itself rejects oversized uploads without requiring the API to receive the bytes.

### 7.4 R2 Bucket Configuration

- Bucket is **public-read** for the `images/` prefix — served via a custom domain (`cdn.yourdomain.com`) mapped to the R2 bucket
- CORS must allow PUT requests from the frontend domain only (not wildcard) on the presigned upload path
- Object deletion: when a `ProjectImage` record is deleted via the API, the corresponding R2 object must also be deleted. This is the responsibility of the `ProjectImagesService` — call the R2 S3-compatible `DeleteObject` API after the database record is removed.

## 9. Tech Stack Suggestion & Approval Flow

1. A developer calls `POST /tech-stack/suggest` with `{ name, category }` while authenticated
2. The service checks case-insensitively whether a `tech_stack` record with the same name already exists in any status
   - If it exists and is `approved` → return `409`: "This technology already exists."
   - If it exists and is `pending` → return `409`: "This technology has already been suggested and is awaiting review."
   - If it exists and is `rejected` → return `409`: "This technology was reviewed and is not available. Contact support if you believe this is an error."
3. If no match, create the record with `status = pending`
4. Return `201 Created` with a message: "Your suggestion has been submitted for review."
5. An admin reviews pending suggestions via `GET /admin/tech-stack/pending`
6. Admin approves via `PATCH /admin/tech-stack/:id/approve` → sets `status = approved`
7. Admin rejects via `PATCH /admin/tech-stack/:id/reject` → sets `status = rejected`
8. Once approved, the technology appears in `GET /tech-stack` and is available for developers to use when tagging projects or adding skills

## 10. Rate Limiting

Rate limiting is applied to **public endpoints only** using `@nestjs/throttler`.

| Setting                 | Value                                             |
| ----------------------- | ------------------------------------------------- |
| Window                  | 1 minute                                          |
| Max requests per IP     | 60                                                |
| Scope                   | Public endpoints only (`/u/*`, `GET /tech-stack`) |
| Authenticated endpoints | Not rate-limited (protected by JWT instead)       |

When the limit is exceeded, return `429 Too Many Requests` with a `Retry-After` header indicating when the window resets.

Apply the throttler guard selectively using a custom decorator rather than globally, to avoid affecting authenticated dashboard API calls.

## 11. Error Handling

### 11.1 Global Exception Filter

A global `HttpExceptionFilter` catches all exceptions and formats them into the standard error envelope:

```json
{
  "success": false,
  "error": {
    "code": "NOT_FOUND",
    "message": "Project not found",
    "details": []
  }
}
```

### 11.2 Error Codes

| HTTP Status | Code                     | When Used                                           |
| ----------- | ------------------------ | --------------------------------------------------- |
| 400         | `VALIDATION_ERROR`       | DTO validation failure                              |
| 400         | `BAD_REQUEST`            | Business logic rejection (e.g. expired reset token) |
| 401         | `UNAUTHORIZED`           | Missing or invalid JWT, wrong credentials           |
| 403         | `FORBIDDEN`              | Authenticated but accessing another user's resource |
| 404         | `NOT_FOUND`              | Resource does not exist or is soft-deleted          |
| 409         | `CONFLICT`               | Duplicate email, username, skill, tech stack        |
| 413         | `PAYLOAD_TOO_LARGE`      | File exceeds size limit                             |
| 415         | `UNSUPPORTED_MEDIA_TYPE` | Invalid file type                                   |
| 429         | `RATE_LIMITED`           | Public API rate limit exceeded                      |
| 500         | `INTERNAL_ERROR`         | Unhandled server error                              |

### 11.3 Prisma Error Handling

The `PrismaService` or a dedicated interceptor must catch Prisma-specific errors and translate them into appropriate HTTP exceptions before they reach the global filter:

- `P2002` (unique constraint violation) → `409 Conflict`
- `P2025` (record not found) → `404 Not Found`
- `P2003` (foreign key constraint failure) → `400 Bad Request`

## 12. Validation Rules

### 12.1 Password

- Minimum 8 characters, maximum 72 characters
- Must contain at least one uppercase letter, one lowercase letter, one digit
- Validated via class-validator on `RegisterDto` and `ResetPasswordDto`

### 12.2 Email

- Valid email format (RFC 5322, validated via `@IsEmail()`)
- Normalized to lowercase before storage and lookup

### 12.3 Username

- 3–30 characters
- Lowercase alphanumeric and hyphens only: `/^[a-z0-9][a-z0-9-]{1,28}[a-z0-9]$/`
- Cannot start or end with a hyphen
- Cannot contain consecutive hyphens
- Case-insensitively unique
- Reserved words (must not be registered as usernames):

```
admin, api, www, mail, support, help, about,
contact, login, register, dashboard, me,
public, static, assets, blog, docs, status
```

### 12.4 Project Slug

- 3–60 characters
- Lowercase alphanumeric and hyphens only
- Same character rules as username
- Unique per user (not globally)
- Auto-generated from title on create; if the generated slug conflicts, append `-2`, `-3`, etc.

### 12.5 URLs

- `liveUrl`, `repoUrl`, and all `ProjectLink.url` values must pass `@IsUrl()` validation if provided
- Protocol must be `http` or `https` — no bare domains accepted

### 12.6 Field Length Limits

| Field                      | Max Length        |
| -------------------------- | ----------------- |
| `user.fullName`            | 100 characters    |
| `user.bio`                 | 500 characters    |
| `project.title`            | 120 characters    |
| `project.shortDescription` | 300 characters    |
| `project.longDescription`  | 50,000 characters |
| `experience.company`       | 100 characters    |
| `experience.role`          | 100 characters    |
| `experience.description`   | 2,000 characters  |
| `projectLink.label`        | 60 characters     |
| `projectImage.altText`     | 200 characters    |
| `techStack.name`           | 60 characters     |

## 13. Non-Functional Requirements

### 13.1 Performance

- Public endpoints must respond in under **500ms** under normal load, excluding database cold-start latency from Neon's serverless compute
- All queries on `projects` filtered by `userId` and `status` must use the composite index `@@index([userId, status, deletedAt])`

### 13.2 Security

- All passwords stored as bcrypt hashes, cost factor 12
- All refresh tokens stored as bcrypt hashes, cost factor 10
- All password reset tokens stored as SHA-256 hashes
- JWT signed with `HS256` using a secret of at least 32 characters
- No sensitive fields (`passwordHash`, `tokenHash`) are ever included in any API response — use Prisma's `select` or `omit` to exclude them explicitly, not accidentally
- CORS must be configured to allow requests only from the frontend origin (`FRONTEND_URL`), not wildcard, for all authenticated endpoints

### 13.3 Reliability

- All cascade deletes are handled at the database level via Prisma's `onDelete: Cascade` — do not implement cascade logic in the service layer
- The one exception is R2 object deletion, which must be handled explicitly in `ProjectImagesService` and `UsersService` (avatar cleanup on account deletion) since R2 is outside the database transaction

### 13.4 Logging

- All unhandled exceptions must be logged with stack traces
- Failed email sends must be logged with the recipient address (masked beyond the first 3 characters) and error message
- Do not log raw tokens, passwords, or full email addresses in production
