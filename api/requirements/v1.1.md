## 1. Project Overview

DevFolio CMS is a multi-user backend platform where software developers register an account, manage their professional portfolio data (projects, case studies, skills, work experience), and expose that data through a public, unauthenticated API. The public API is consumed by a developer's own external portfolio website or any other client they choose to point at it.

The backend is responsible for:

- Developer account management and authentication
- Full CRUD operations for all portfolio entities
- Serving a clean, public, read-only API scoped by username
- Handling image uploads directly through the backend to Cloudflare R2, atomically with database writes
- Sending transactional emails via Nodemailer + SMTP
- Managing a shared, admin-governed technology lookup list

There is no API key system. The public API requires no authentication. Draft and archived content is never reachable from outside the authenticated dashboard.

### 2.2 Repository Structure

The backend lives in an `api/` directory inside a Bun-based monorepo managed by Turborepo:

```
root/
├── api/          ← NestJS backend (this document)
├── client/       ← Next.js frontend (separate requirements)
├── packages/     ← Shared TypeScript types/configs
├── package.json  ← Workspace root
└── turbo.json
```

### 2.3 Architecture Principles

- **Modular structure:** each entity lives in its own NestJS module with its own controller, service, and DTOs.
- **No business logic in controllers:** controllers handle HTTP concerns only; all logic lives in services.
- **No raw SQL:** all database access goes through Prisma.
- **Soft deletes for users, projects, and case studies:** records are never hard-deleted; a `deleted_at` timestamp is set instead, and all queries filter on `deleted_at IS NULL`.
- **Backend-handled uploads:** the backend receives image bytes directly via `multipart/form-data`, uploads them to R2, and saves the database record atomically. The client never communicates with R2 directly.
- **Atomic image + data writes:** image upload to R2 and the corresponding database write succeed together or fail together, using compensating deletes on R2 if the database transaction fails.
- **Global exception filter:** all unhandled errors return a consistent JSON error shape.
- **Global validation pipe:** all incoming request bodies are validated via class-validator DTOs before reaching the controller.

## 3. Database Schema

All tables use UUID primary keys generated by the application layer (not the database). All timestamps are stored in UTC.

### 3.1 users

```prisma
model User {
  id           String    @id @default(uuid())
  email        String    @unique
  passwordHash String    @map("password_hash")
  username     String    @unique
  fullName     String?   @map("full_name")
  avatarUrl    String?   @map("avatar_url")
  bio          String?

  createdAt    DateTime  @default(now()) @map("created_at")
  updatedAt    DateTime  @updatedAt @map("updated_at")
  deletedAt    DateTime? @map("deleted_at")

  projects       Project[]
  caseStudies    CaseStudy[]
  skills         UserSkill[]
  experience     Experience[]
  refreshTokens  RefreshToken[]
  passwordResets PasswordReset[]

  @@map("users")
}
```

**Constraints:**

- `email` — unique, required, valid email format, normalized to lowercase
- `username` — unique, required, 3–30 characters, lowercase alphanumeric and hyphens only, cannot start or end with a hyphen, cannot be a reserved word (see Section 12.3)
- `passwordHash` — never exposed via any API response
- `deletedAt` — all queries must filter `deletedAt IS NULL` unless explicitly performing a recovery operation

### 3.2 refresh_tokens

```prisma
model RefreshToken {
  id        String    @id @default(uuid())
  userId    String    @map("user_id")
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  tokenHash String    @unique @map("token_hash")
  expiresAt DateTime  @map("expires_at")
  revokedAt DateTime? @map("revoked_at")
  createdAt DateTime  @default(now()) @map("created_at")

  @@map("refresh_tokens")
}
```

**Notes:**

- Refresh tokens are stored as bcrypt hashes, never plaintext
- Revoked tokens are soft-revoked (`revokedAt` set), not deleted, for audit purposes
- Expired and revoked tokens should be purged by a scheduled cleanup job (v1: manual or cron, not a hard requirement)

### 3.3 password_resets

```prisma
model PasswordReset {
  id        String    @id @default(uuid())
  userId    String    @map("user_id")
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  tokenHash String    @unique @map("token_hash")
  expiresAt DateTime  @map("expires_at")
  usedAt    DateTime? @map("used_at")
  createdAt DateTime  @default(now()) @map("created_at")

  @@map("password_resets")
}
```

**Notes:**

- Reset tokens expire after 1 hour
- Tokens are single-use: `usedAt` is set on redemption and the token is rejected on any subsequent use
- Token is stored as a SHA-256 hash; the plaintext token is sent to the user's email only and never persisted

### 3.4 projects

```prisma
model Project {
  id               String        @id @default(uuid())
  userId           String        @map("user_id")

  repoUrl          String?       @map("repo_url")

enum ProjectType {
```

### 3.5 case_studies

```prisma
model CaseStudy {
  id           String          @id @default(uuid())
  userId       String          @map("user_id")
  user         User            @relation(fields: [userId], references: [id], onDelete: Cascade)
  projectId    String?         @unique @map("project_id")
  project      Project?        @relation(fields: [projectId], references: [id], onDelete: SetNull)

  title        String
  slug         String
  excerpt      String?
  coverImageUrl String?        @map("cover_image_url")
  coverImageKey String?        @map("cover_image_key")

  overview     String?
  problem      String?
  approach     String?
  solution     String?
  results      String?

  status       CaseStudyStatus @default(draft)
  isFeatured   Boolean         @default(false) @map("is_featured")
  displayOrder Int             @default(0) @map("display_order")

  createdAt    DateTime        @default(now()) @map("created_at")
  updatedAt    DateTime        @updatedAt @map("updated_at")
  deletedAt    DateTime?       @map("deleted_at")

  @@unique([userId, slug])
  @@index([userId, status, deletedAt])
  @@map("case_studies")
}

enum CaseStudyStatus {
  draft
  published
  archived
}
```

**Constraints:**

- `slug` — unique per `userId`, same rules as project slug, auto-generated from `title` on creation
- `projectId` — optional FK to `projects`. If the linked project is deleted, `projectId` is set to `null` (`onDelete: SetNull`) — the case study is not deleted along with the project
- `projectId` has a `@unique` constraint — one project can be linked to at most one case study
- `excerpt` — max 500 characters, shown in case study list/cards
- `coverImageKey` — the R2 object key, stored alongside the public URL to allow deletion of the R2 object when the case study is deleted or cover replaced
- `overview`, `problem`, `approach`, `solution`, `results` — each is a markdown string, max 10,000 characters per section
- Only `published` case studies appear in the public API
- Soft-deleted case studies are excluded from all public and private reads

## 5. Authentication & Security

### 5.1 Registration Flow

1. Validate the request body via `RegisterDto`
2. Check `email` is not already registered (case-insensitive) — return `409 Conflict` if taken
3. Check `username` is not already taken (case-insensitive) — return `409 Conflict` if taken
4. Validate `username` against allowed character rules and reserved word list (Section 12.3)
5. Hash the password using `bcrypt` with a cost factor of **12**
6. Create the user record
7. Return the user's public profile (no tokens — require explicit login after registration)

### 5.2 Login Flow

1. Find user by email (case-insensitive) — return `401 Unauthorized` with a generic message if not found (do not reveal whether the email exists)
2. Check `deletedAt IS NULL` — return `401` if account is soft-deleted
3. Compare submitted password against stored hash using `bcrypt.compare`
4. On success, generate:
   - **Access token:** JWT signed with `JWT_SECRET`, payload `{ sub: userId, username }`, expires in **15 minutes**
   - **Refresh token:** cryptographically random string (32 bytes, hex-encoded), hashed with bcrypt (cost 10) and stored in `refresh_tokens`, expires in **14 days**
5. Return both tokens in the response body

### 5.3 Token Refresh Flow

1. Accept the refresh token from the request body
2. Hash the submitted token and look up a matching, non-revoked, non-expired `RefreshToken` record
3. Verify the associated user exists and is not soft-deleted
4. **Rotate the token:** revoke the used refresh token and issue a new one
5. Issue a new access token
6. Return both new tokens

### 5.4 Logout Flow

1. Require a valid access token
2. Accept the current refresh token from the request body
3. Set `revokedAt` on the matching `RefreshToken` record
4. Return `200 OK`

### 5.5 Forgot Password Flow

1. Accept an email address
2. Look up the user by email — **always return the same generic success response regardless of whether the email is registered** (prevents email enumeration)
3. If the user exists and is not soft-deleted:
   - Invalidate any existing unused password reset tokens for this user by setting `usedAt = now()`
   - Generate a cryptographically random reset token (32 bytes, hex-encoded)
   - Store a SHA-256 hash of the token in `password_resets` with `expiresAt = now() + 1 hour`
   - Send the reset email via Nodemailer (Section 8)
4. Return `200 OK` with a generic message: `"If that email is registered, you will receive a reset link shortly"`

### 5.6 Reset Password Flow

1. Accept `token` and `newPassword` from the request body
2. Hash the submitted token with SHA-256 and look up a matching `PasswordReset` record
3. Reject if: token not found, `usedAt` is already set, or `expiresAt` is in the past — return `400 Bad Request`
4. Validate `newPassword` against password rules (Section 12.1)
5. Hash the new password with bcrypt (cost 12)
6. Update `user.passwordHash`
7. Mark the reset token as used: `usedAt = now()`
8. Revoke all existing refresh tokens for this user (force re-login everywhere)
9. Return `200 OK`

### 5.7 Password Rules

- Minimum 8 characters, maximum 72 characters
- At least one uppercase letter, one lowercase letter, one digit

### 5.8 Authorization

- All authenticated endpoints use a `JwtAuthGuard` validating the access token from `Authorization: Bearer <token>`
- Controllers use a `@CurrentUser()` decorator to extract `userId` from the JWT payload
- All service methods operating on owned resources must verify `userId` from the token matches the resource's `userId` — return `403 Forbidden` if not

| DELETE | `/me/projects/:id/images/:imageId` | JWT  | Remove image record and delete R2 object        |
| PATCH  | `/me/projects/:id/images/reorder`  | JWT  | Reorder all images in one call                  |

**POST /me/projects/:id/images** accepts `multipart/form-data` with `images[]` files. For each file, the service uploads to R2 and saves a `ProjectImage` record. The entire operation (all files + all records) is atomic — if any upload or any DB write fails, all successfully uploaded R2 objects are deleted and no records are saved.

**PATCH /me/projects/:id/images/:imageId** with `isCover: true`: the service sets `isCover = false` on all other images for this project in the same Prisma transaction before setting the new one to `true`.

**DELETE /me/projects/:id/images/:imageId**: deletes the `ProjectImage` record from Postgres and calls R2 `DeleteObject` using the stored `key`. If R2 deletion fails, log the error but still return success — the database record is the source of truth for what is "shown," and the orphaned R2 object can be cleaned up manually.

## 7. File Upload (Cloudflare R2)

### 7.1 Upload Architecture

The backend receives image files directly via `multipart/form-data`. The client never communicates with R2 directly. All uploads go through the `UploadsService`.

```
Client                      NestJS API                  Cloudflare R2
  |                              |                            |
  |-- POST /me/projects -------> |                            |
  |   multipart/form-data        |                            |
  |   { title, images[] }        |                            |
  |                              |-- validate files           |
  |                              |-- PutObject(image1) -----> |
  |                              |-- PutObject(image2) -----> |
  |                              |                            |
  |                              |-- prisma.$transaction {    |
  |                              |     create project         |
  |                              |     createMany images      |
  |                              |   }                        |
  |                              |                            |
  |                              | if tx fails:               |
  |                              |-- DeleteObject(image1) --> |
  |                              |-- DeleteObject(image2) --> |
  |                              |                            |
  |<-- 201 Created ------------- |                            |
```

### 7.2 Atomicity Pattern

```typescript
async createProjectWithImages(userId, dto, files) {
  // Step 1: validate all files before touching DB or R2
  for (const file of files) {
    this.uploadsService.validateFile(file);
  }

  // Step 2: upload all images to R2, collect keys + URLs
  const uploaded: { key: string; url: string }[] = [];

  try {
    for (const file of files) {
      const key = this.uploadsService.generateKey(userId, 'project_image', file);
      await this.uploadsService.putObject(key, file);
      uploaded.push({ key, url: `${R2_PUBLIC_URL}/${key}` });
    }
  } catch (r2Error) {
    // R2 upload failed — clean up any objects that did upload
    await this.uploadsService.deleteObjects(uploaded.map(u => u.key));
    throw new InternalServerErrorException('Image upload failed');
  }

  // Step 3: save project + image records in one Prisma transaction
  try {
    return await this.prisma.$transaction(async (tx) => {
      const project = await tx.project.create({ data: { userId, ...dto } });
      await tx.projectImage.createMany({
        data: uploaded.map((u, i) => ({
          projectId: project.id,
          url: u.url,
          key: u.key,
          isCover: i === dto.coverImageIndex,
          displayOrder: i,
        })),
      });
      return project;
    });
  } catch (dbError) {
    // DB failed — compensate by deleting all uploaded R2 objects
    await this.uploadsService.deleteObjects(uploaded.map(u => u.key));
    throw new InternalServerErrorException('Failed to save project');
  }
}
```

The same pattern applies to case study cover image uploads.

### 7.3 Object Key Naming

```
images/{userId}/project_image/{uuid}.{ext}
images/{userId}/avatar/{uuid}.{ext}
images/{userId}/case-study-covers/{uuid}.{ext}
```

### 7.4 File Constraints

| Property                        | Constraint                              |
| ------------------------------- | --------------------------------------- |
| Allowed content types           | `image/jpeg`, `image/png`, `image/webp` |
| Maximum file size               | **5 MB per file**                       |
| Maximum images per project      | **10**                                  |
| Maximum case study cover images | **1**                                   |

Content type is validated from the file's MIME type as parsed by multer — do not trust the filename extension alone.

### 7.5 R2 Configuration

- Bucket is **public-read** for the `images/` prefix, served via a custom domain (`R2_PUBLIC_URL`)
- CORS must allow PUT requests from the frontend origin only, not wildcard
- The `key` field is stored on every image record to enable targeted R2 object deletion
- The `UploadsService` exposes a `deleteObjects(keys: string[])` method used by both the atomicity pattern and the image delete endpoints

## 9. Tech Stack Suggestion & Approval Flow

1. Developer calls `POST /tech-stack/suggest` with `{ name, category }`
2. Service checks case-insensitively whether a record with the same name exists in any status:
   - `approved` → `409`: "This technology already exists."
   - `pending` → `409`: "This technology has already been suggested and is awaiting review."
   - `rejected` → `409`: "This technology was reviewed and is not available. Contact support if you believe this is an error."
3. No match → create record with `status = pending`, return `201 Created`
4. Admin reviews via `GET /admin/tech-stack/pending`
5. Admin approves → `status = approved`, technology becomes available in `GET /tech-stack`
6. Admin rejects → `status = rejected`

## 13. Non-Functional Requirements

### 13.1 Performance

- Public endpoints must respond in under **500ms** under normal load, excluding Neon cold-start latency
- Composite indexes on `[userId, status, deletedAt]` on both `projects` and `case_studies`

### 13.2 Security

- Passwords: bcrypt, cost factor 12
- Refresh tokens: bcrypt hashes, cost factor 10
- Password reset tokens: SHA-256 hashes
- JWT: HS256, secret minimum 32 characters
- No sensitive fields (`passwordHash`, `tokenHash`, `key`) ever included in API responses
- CORS restricted to `FRONTEND_URL` only, not wildcard

### 13.3 Reliability

- All cascade deletes handled at the database level via `onDelete: Cascade`
- R2 object deletion handled explicitly in service layer (not by DB cascade) for: project image delete, case study cover delete/replace, avatar replace, account delete
- If R2 deletion fails on a delete operation, log the error and return success — the DB record is the source of truth for what is displayed

### 13.4 Logging

- All unhandled exceptions logged with stack traces
- Failed email sends logged with masked recipient address and error message
- Never log raw tokens, passwords, or full email addresses in production
